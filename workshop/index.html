星源 AgentKit企业级 Agent Infra 平台

一句话定位：当 Agent 从 Demo、原型走向真实业务或商业化服务，AgentKit 负责把它稳定、安全、可观测地运行起来。

适用场景：内部业务 Agent 的工具链托管，以及商业化服务 Agent 的云上高可用、弹性运行和安全隔离。

AgentKit部署 · 连接 · 运行 · 治理
部署把 Agent 托管到平台，管理版本、环境和生命周期。
连接统一接入知识库、记忆库、MCP 工具和业务系统。
运行提供 Agent Runtime、Sandbox 与弹性资源底座。
治理记录 Trace、评测效果、控制权限和执行边界。

为什么 Agent 需要 Infra？

一个 Agent 做 Demo 不难，真正进入业务使用或对外服务时，难点不在“能不能回答”，而在能否稳定连接系统、执行任务、隔离风险、持续观测和迭代。

连接难Agent 需要访问知识库、内部 API、业务系统和第三方工具。如果每个团队各自适配，鉴权、参数转换、错误处理都会重复建设。
运行难Agent 可能要长期运行、并发处理任务、执行代码或调用 Skill。没有统一运行底座，就容易依赖临时机器、脚本和人工运维。
安全难高自主性的 Agent 可能执行代码、访问文件、调用外部接口，需要沙箱隔离、权限控制和边界约束。
治理难上线后如果效果不稳定、调用失败或成本异常，需要能够回看调用链路、评测效果、定位问题并持续优化。

AgentKit 是什么？

AgentKit 是面向企业级 Agent 应用的基础设施平台。它不替使用方决定 Agent 该怎么思考、怎么写业务逻辑，而是提供上线和服务化所必需的平台能力。

01

不是应用开发平台

Agent 的策略、流程和业务逻辑仍由使用方掌握。

02

是运行托管平台

负责 Agent 的托管、调试、发布、弹性运行和生命周期管理。

03

是能力连接平台

把知识库、记忆库、MCP、Skills、内部 API 等能力统一接入。

04

是治理观测平台

让 Agent 的调用链、效果、成本、权限和执行边界可被管理。

哪些 Agent 场景适合接入？

接入 AgentKit 的关键，不是 Agent 有多会聊天，而是它是否需要连接知识、工具、执行环境和治理能力。下面用三类典型 Agent 场景说明。

Reactive / Tool-Calling

响应型智能体

面向控制台小助手、实时查询机器人等轻量场景，根据用户当前输入选择知识或工具并返回结果。

使用模块
Agent Runtime知识库MCP 工具AI 网关
结构形态

用户问题 → Runtime 托管 Agent → 知识库检索 / MCP 工具调用 → 模型汇总 → 返回答案。

业务逻辑

适合把产品文档、控制台状态、资源信息和常用 API 接入 Agent，让用户用自然语言完成查询、解释和简单操作建议。

Workflow / Layered Pattern

工作流引导型智能体

面向 HR 招聘助手、工单处理、合规初审等流程型场景，把 Agent 决策放进明确的业务步骤中。

使用模块
Agent RuntimeSkillsMCP 工具Trace / Eval
结构形态

流程入口 → Router 分流 → SOP 状态机 → Skills / MCP 执行业务动作 → Trace 记录与效果评估。

业务逻辑

以招聘助手为例，可以串联 JD 解析、简历筛选、候选人匹配、面试问题生成和结果沉淀，业务边界清晰、链路可追踪。

Autonomous / Plan-and-Solve

自主规划型智能体

面向灵犀 Claw、KSCC 等复杂任务 Agent，用户只给出目标，Agent 自主拆解任务、调用工具并动态修正计划。

使用模块
Agent RuntimeSandboxSkills记忆库Trace / Eval
结构形态

目标输入 → Planner 生成任务树 → Sandbox 执行代码/探测环境 → Skills 调用能力 → 自我反思与计划修正。

业务逻辑

适合研发辅助、复杂资料调研、自动化验证等高自主性任务。AgentKit 重点提供隔离运行、能力复用、长期上下文和全链路观测。

如果想接入，可以怎么开始？

不同团队的起点不一样：可以从 AI 辅助开发开始，也可以把已有 Agent 迁移到平台，正式业务则可以进入完整方案设计。

非技术同学

用 Coding 工具启动原型

  • 用 Cursor / Codex / KSCC(Claude code）描述 Agent 场景。
  • 结合 KSADK 文档生成 Agent 代码与部署步骤。
  • 按 AgentKit 文档接入知识库、MCP 工具和 Skills。
技术同学 / 已有 Agent

把已有服务迁移到 AgentKit

  • 梳理现有 Agent 框架、依赖和工具调用。
  • 基于 KSADK 支持框架接入，或打包为容器镜像。
  • 迁移到 AgentKit 后复用运行、连接和观测能力。
正式业务需求

进入云上方案设计

  • 适合线上业务、客户交付或大规模 Agent 服务。
  • 联系销售、售前或产品同学梳理架构与资源规格。
  • 输出部署架构、接入路径、容量规划和报价方案。
接入方式可以从小原型开始，也可以从正式业务方案开始；AgentKit 承接的是 Agent 上线后的运行、连接、治理和云上服务化能力。

内部业务接入案例

以一个内部业务 Agent 为例，可以看到 AgentKit 如何把已有 Agent 原型接入平台，完成能力连接、运行托管、安全执行和持续观测。

Step 1业务 Agent 原型业务团队已有 Agent 想法或原型，明确它要完成的任务。
Step 2能力连接梳理需要接入的知识库、API、MCP 工具或 Skills。
Step 3运行托管通过 AgentEngine 托管运行，配置环境、版本和资源。
Step 4安全执行需要代码或工具执行时，进入 Sandbox 隔离运行。
Step 5观测优化通过 Trace / Eval 查看效果、链路和成本，持续迭代。
